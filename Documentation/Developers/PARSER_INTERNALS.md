# Parser Internals

> **Scope:** `rune::parser` compiler library  
> **Location:** `Code/Rune/Parser/`

This document describes the internals of the Rune parser, which is responsible for turning a flat sequence of tokens into a structured Abstract Syntax Tree (AST). It is the second stage of the compiler front-end, sitting between the Lexer and Semantic Analysis (Sema).

## Overview

The `rune::parser` library implements a predictive recursive descent parser. It consumes tokens generated by `rune::lexer` and produces a syntax tree.

The parser is purely syntactic. It records the grammatical structure of the code but assigns no semantic meaning. It **does not** resolve names, verify type safety, check borrowing rules, or load external modules. All of those responsibilities belong to `rune::sema`.

## The Parse Pipeline

While the `Parser` class itself typically operates on a single token stream, real Rune projects consist of multiple source files. This is coordinated by `ParsePipeline.hpp`:

- **`parseSourceFiles`**: Takes a list of `.rune` files, invokes the lexer on each one independently, and parses them.
- **Merging**: The individual file parse results are merged into a single root `parser::Program` node.
- **Tracking**: To ensure diagnostics and Semantic Analysis know where each declaration came from, top-level AST nodes are tagged with a `SourceFileID`.

## AST Architecture

The Abstract Syntax Tree is defined in `AST.hpp` and `AST.cpp`.

### Node Hierarchy
All AST nodes derive from a common base, which branches into three main functional categories:
1. **`Declaration`**: Top-level structures like `func`, `struct`, `let`, `var`, `enum`, and `interface`.
2. **`Statement`**: Control flow and block-level constructs like `if`, `while`, `return`, `defer`, and `match`.
3. **`Expression`**: Constructs that produce values, like binary operations (`a + b`), function calls, array literals, and identifiers.

### The Visitor Pattern
The AST leverages the Visitor pattern (via `accept()` methods). This allows external compiler passes—like `ASTContext` declaration registration in Sema, type checking, and bytecode emission—to traverse the tree strictly decoupled from the AST node definitions.

### Type Expressions
Types written in the source code (e.g., `Array<Int>`) are parsed as `TypeExpression` nodes. The parser does not know if `Array` is a struct, an alias, or if it even exists. Lowering these `TypeExpression` syntactic nodes into canonical semantic types is handled later by `TypeResolution` in Sema.

## Parsing Strategy

The parser is a traditional recursive descent parser, navigating through grammar rules using standard techniques.

### Cursor and Token Matching
The parser maintains a cursor over the token array. It uses several helper methods:
- **`peek()`**: Looks at the current token without consuming it.
- **`match(TokenType)`**: Checks if the current token matches a specific kind. If it does, it consumes it and returns true.
- **`expect(TokenType, ...)` / `consume(...)`**: Asserts that the current token is a specific kind and consumes it. If not found, it reports an error.

### Precedence and Expressions
Expressions are typically parsed using Pratt parsing (or precedence climbing). This handles operator precedence and associativity organically, ensuring that `a + b * c` correctly parses as `a + (b * c)` based on precedence levels.

## Attributes and Modifiers

The parser is responsible for collecting modifiers and attributes, and attaching them to the relevant AST nodes:

- **Access Modifiers**: `public`, `internal`, `private` are parsed and stored directly on `Declaration` nodes.
- **Attributes**: `@builtin`, `@symbol`, etc., are collected into attribute lists and attached to declarations.
- **Generic Arity**: The parser recognizes `<T>` on structs or functions and stores the generic parameters as syntax, without enforcing semantic instantiation rules.

## Error Handling and Recovery

When the parser encounters unexpected tokens, it uses the `rune::DiagnosticEngine` to emit a user-facing error using the exact `SourceLocation` provided by the Lexer.

Instead of immediately aborting, the parser attempts **synchronization**. When an error occurs, it discards tokens until it finds a reliable synchronization point—typically a statement boundary (like a newline or `;`) or the start of a new declaration (like `func` or `struct`). This allows the parser to recover and continue parsing, reporting as many syntax errors as possible in a single compilation run.

## Integration with Sema

Once `parseSourceFiles` completes successfully, it returns a single `parser::Program` node. This root node represents the entire multi-file module and is passed into `prepareModule(ASTContext&, parser::Program const&)`, kicking off the semantic analysis phases.
